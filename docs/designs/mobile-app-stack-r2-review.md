# Review of mobile-app-stack-r2.md against the server codebase

Server-side verification of the claims in `mobile-app-stack-r2.md` (authored
by the mobile-app agent), dated 2026-07-23. Each claim was checked against the
actual v0.6.0 code, not the design docs. Corrections first, then the items
this review closed, then the verified-accurate remainder.

## Corrections to r2

### 1. The ntfy/UnifiedPush dispatcher is ALREADY SHIPPED — r2 says it "remains"

r2 states (twice: §Push and Backend-prerequisites item 5) that "the dispatcher
wiring remains". Wrong: v0.6.0 ships the full server half.

- `services/push.py` — `send_push_to_profile` POSTs the notification JSON to
  every active UnifiedPush device (5s timeout, consecutive-failure counting,
  auto-revoke after 10 strikes); FCM rows accepted at registration, skipped at
  dispatch (as designed, pending the `play` flavor).
- `tasks.dispatch_native_push` (Celery) + a `NotificationLog` post_save
  receiver (`models/notifications/signals.py`, `enqueue_native_push`) mean
  **every existing notification call site already dispatches native push** —
  nothing further is needed server-side for the client to start receiving.

**Client contract the app must implement** (r2 doesn't specify it): the
registered endpoint receives `POST` with a JSON body of the same payload the
WebSocket bell uses (`as_push_payload`):
`{"id", "title", "message" (truncated to 300 chars), "url",
"notification_type", "importance"}`. For a stock ntfy server, that means the
app should register a UnifiedPush endpoint URL (`/<topic>` with UP enabled),
not rely on ntfy's title/priority header conventions — the server sends one
JSON document, transport-agnostically.

### 2. Only registration-time SSRF validation protects push endpoints — a client-visible constraint r2 omits

`register_device` rejects endpoints that are non-http(s), carry embedded
credentials, or resolve to private/loopback/link-local/reserved ranges. A
self-hosted ntfy on a LAN address will therefore be **rejected at
registration**. That is a deliberate v1 SSRF stance (documented in
`services/push.py`), but the app's settings UI should surface the error
message rather than treat registration as infallible.

### 3. Desktop OAuth redirect needs no fixed port — unaddressed in r2

Verified in django-oauth-toolkit 3.3.0 (`redirect_to_uri_allowed`): loopback
IP redirect URIs match **port-insensitively** per RFC 8252 §7.3. The
provisioned registration (below) includes `http://127.0.0.1/callback`, so the
desktop app may bind any free port. Custom-scheme and loopback are both
covered by tests (`test_oauth_client_provisioning.py`).

## Gaps r2 correctly identified — now closed (this session)

### 4. E2EE byte-exact fixture — produced and independently verified

- **`docs/e2ee-interop-fixture.json`** — generated by the *actual web-client
  module* (`shared/e2ee-crypto.ts` under Bun/libsodium WASM) via the committed
  generator `frontend/ts/tools/generate-e2ee-fixture.ts`. Covers: both Argon2id
  derivations (expected keys included), the wrapped private key, the
  recovery-key wrap AND its grouped-base32 display encoding, a sealed
  conversation key, and one encrypted message. The `_readme` inside is the
  verification contract, including the asymmetry that bites: key-wrap blobs
  are `nonce‖box` combined, message nonce/ciphertext are stored separately.
- Cross-validated byte-for-byte with PyNaCl (independent implementation, same
  as the server's own interop tests): all derivations, unwraps, unseal,
  decrypt, display encoding, and pub-from-priv consistency pass.
- **Argon2id parameter mapping for the Dart side** (this is where
  implementations diverge; r2's `cryptography`-package interim especially):
  libsodium `opslimit=2` → iterations t=2; `memlimit=67_108_864` **bytes** →
  m=65536 **KiB**; parallelism p=1 (libsodium fixes lanes at 1); version 1.3
  (`ALG_ARGON2ID13`); output 32 bytes; salt 16 bytes; base64 standard WITH
  padding. If the pure-Dart interim produces a different byte-stream, the
  fixture will say so immediately.

### 5. First-party OAuth2 application — provisioned reproducibly

New management command: `manage.py provision_mobile_oauth_client`
(idempotent on client_id; corrects drifted fields on re-run).

- `client_id`: **`urbanlens-mobile`** (public client — the id is not a secret
  and is safe to bake into the app; there is no client secret).
- Grant: authorization-code, **PKCE required** (global).
- Redirect URIs: `urbanlens://oauth/callback` (Android/iOS custom scheme —
  scheme added to `OAUTH2_PROVIDER["ALLOWED_REDIRECT_URI_SCHEMES"]`) and
  `http://127.0.0.1/callback` (desktop loopback, any port).
- Endpoints: `/oauth/authorize/`, `/oauth/token/`, `/oauth/revoke_token/`.
- Scopes: `profile:read pins:read pins:write push:manage` (default grant).
- Run it per environment (dev VM, production); nothing is stored in the repo
  beyond the command itself.

If the app wants a different scheme/callback, rerun with `--redirect-uri`
(and keep the scheme allowlisted in settings).

### 6. `description` + `pin_type` on external pin create — shipped

`POST pins/` now accepts both (BACKEND_CHANGES §2):

- `description`: free text, capped at 10,000 chars at the API boundary.
- `pin_type`: any `PinType` value (`location`, `parcel`, `building`,
  `entrance`, `poi`, `danger`, `other`); invalid values 400. Supplying one
  sets `pin_type_is_user_provided=True` so automatic building/parcel
  classification never overwrites an explicit field choice — omitting it
  keeps the classifiable `location` default, exactly like a map-UI drop.

  existing per-family ownership checks unchanged.
