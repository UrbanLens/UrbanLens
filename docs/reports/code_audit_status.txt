CODE AUDIT STATUS - tracking doc for docs/reports/unused_functions.txt triage
================================================================================
Working file-by-file, starting with whichever file has the most HIGH-confidence
findings. Each entry records the disposition decided on and what (if anything)
was done about it. Categories:

  UNINCORPORATED  - useful code that exists but isn't wired into the site yet
  DEPRECATED      - superseded by other code; needs removal
  DEAD            - never-useful/no-op code; deleted on sight
  REFACTOR        - kept, but needs cleanup/integration work to be solid
  FALSE-POSITIVE  - report was simply wrong (script blind spot); no action taken

NOTE on a systemic false-positive source found in this codebase: Celery tasks
are queued via `safely_enqueue_task(task_func, ...)` (services/celery.py),
passing the task function as a value rather than calling `task_func.delay(...)`
directly. The unused-functions scanner only counts direct Call nodes, so any
task only ever invoked this way reports 0-1 refs even when it's fully wired up.
Always grep for the bare function name across `src/` before trusting a 0/1-ref
verdict on anything in dashboard/tasks.py or similar dispatch-by-reference code.

src/urbanlens/dashboard/tasks.py  (16 HIGH findings - most in report)
================================================================================
Verified 10 of the 16 flagged tasks are FALSE-POSITIVE: create_location_for_pin,
suggest_location_category, run_user_data_export, cleanup_export_artifacts_task,
run_user_data_import, cleanup_import_artifacts_task, rebuild_map_pin_cache,
prefetch_location_external_data, run_database_backup, run_scheduled_database_backup
- all genuinely enqueued via `safely_enqueue_task(...)` from controllers/services,
just invisible to the scanner.

The remaining 6 had zero references anywhere outside tasks.py itself (confirmed
by direct grep, not just the scanner). Resolved each:

  - apply_admin_code_update - DEAD, DELETED.
    Verbatim duplicate of SiteAdminPullLatestCodeView.post (controllers/site_admin.py:373).
    That view already has a synchronous fetch() call with a spinner + toastr
    success/error UX (templates/dashboard/pages/site_admin_stats.html:198-235) -
    a single dev-only admin button, bounded duration. Converting to Celery would
    require breaking the view's existing synchronous JSON contract (which has
    test coverage in test_site_admin_stats.py) and adding job-status polling,
    for a button that already has adequate UX. Not worth the churn.

  - refresh_smithsonian_images - DEAD, DELETED.
    Called the Smithsonian gateway and discarded the result - there is no cache
    layer for Smithsonian data anywhere in the codebase (controllers/pin.py's
    get_smithsonian_images always fetches live), so "warming" achieved nothing.
    NOTE from Jess: There should be a cache layer for Smithsonian data, as well as all
    API data from any API, in order to reduce api usage and improve performance.

  - refresh_weather_forecast - DEAD, DELETED.
    Same shape as above: OpenWeatherMapGateway (services/openweather/gateway.py)
    has no caching at all (verified by reading the gateway), so the task fetched
    and threw away data. No-op stub.

  - extract_import_archive - DEAD, DELETED (paired with import_pins_from_paths).
  - import_pins_from_paths - DEAD, DELETED.
    Both formed an abandoned disk-path-based import pipeline (read archive from
    a path, extract to a dir, then import from saved files). The live import
    path (PinController.upload_takeout, controllers/pin.py:356) already handles
    archives in-memory and streams progress via SSE - a strictly better UX
    (instant feedback, no polling) than what these two tasks would have offered.
    Nothing in the codebase saves uploads to MEDIA_ROOT and calls these, and no
    tests reference them. Superseded, not unfinished.

  - refresh_pin_web_search - UNINCORPORATED, now INTEGRATED.
    It writes to the *same* cache key
    (`make_cache_key("web_search_pin", pin.pk)`) that PinController.web_search
    (controllers/pin.py:230) reads on the pin detail page. Wired it into the pin
    creation flow in controllers/maps.py, right next to the existing
    prefetch_location_external_data enqueue, so new pins get their web-search
    results pre-warmed the same way Wikipedia/NPS/Google Places already are.

Net result: tasks.py shrank from 433 to ~350 lines (5 dead tasks removed), and
one genuinely-useful-but-disconnected task got wired up.
