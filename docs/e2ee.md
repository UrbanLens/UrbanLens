# End-to-end encryption for direct messages

UrbanLens direct messages are end-to-end encrypted: the message body is
encrypted in the sender's browser and only ever decrypted in a participant's
browser. The server stores opaque ciphertext and cannot read it, and it never
sees a plaintext password or a private key. This document describes the scheme,
the exact blob formats, the recovery flows, and the honest limits.

> **Scope (v1):** only the message **text** is encrypted. Photos, attached
> markup maps, and `@pin`/`@trip`/`@friend` share cards remain server-visible
> (they are metadata-heavy and reuse existing server-side pipelines). Reactions,
> read receipts, typing, presence, and "who talked to whom, when, and how big"
> are metadata and stay visible to the server.

## Threat model

**Protects against:** a stolen database dump or a passive read of stored data.
An attacker with the full database sees ciphertext, wrapped private keys, and
Argon2id-hashed credentials — none of which decrypt messages without a user's
password or recovery key.

**Does NOT protect against:** a malicious or compromised server *operator*. The
server ships the JavaScript that does the encryption, so an operator who serves
backdoored JS can capture plaintext. This is inherent to all web-delivered E2EE
(Proton Mail, WhatsApp Web, etc.). Native apps with pinned code would be needed
to close this gap.

**Also out of scope:** forward secrecy. There is no ratchet — a compromised
private key decrypts that user's whole history. This is a deliberate trade for
**recoverability**: UrbanLens explicitly does not want Signal's "lose your
device, lose your history" failure mode.

## Keys and how they relate

```
password ──Argon2id(auth_salt)──▶ authKey ──▶ (sent to server; Django hashes it as the "password")
        └─Argon2id(wrap_salt)──▶ wrapKey ──▶ encrypts ─┐
                                                        ├─▶ identity private key (X25519)
recovery key (32 random bytes) ─────────── encrypts ───┘
                                                        │
identity keypair ──▶ public key ──▶ seals ──▶ conversation key (random 32 bytes, per pair)
                                                        │
conversation key ──▶ encrypts ──▶ message body
```

- **Identity keypair** — one X25519 (`crypto_box`) keypair per user, generated
  in the browser at enrollment. The public key is stored in the clear; the
  private key is stored only in *wrapped* form.
- **`authKey` vs `wrapKey`** — both are Argon2id derivations of the login
  password, but with **independent random salts**. `authKey` is sent to the
  server as the login credential (Django hashes it again with its normal
  password hasher). `wrapKey` never leaves the browser and encrypts the private
  key. Because the salts are independent, the server — which learns `authKey`
  and stores its hash — cannot derive `wrapKey`.
- **Recovery key** — 32 random bytes shown to the user once at enrollment
  (grouped base32, e.g. `R3K7-Q2ND-…`). It wraps a *second* copy of the private
  key. This is the unwrap path for OAuth users on a new device and for password
  users after a password reset on a cold device.
- **Conversation key** — a random 32-byte symmetric key per conversation pair,
  generated by whoever sends first. It is sealed (`crypto_box_seal`) once to
  each participant's public key and stored server-side as two opaque blobs.
  Message bodies are encrypted with `crypto_secretbox` under it.
- **Group key** — the group-chat analogue: a random 32-byte key per
  `(group, version)`, sealed once per member (`GroupKeyEnvelope`). A group is
  encrypted iff **every** active member has a key bundle. Clients mint a new
  version whenever the latest one no longer covers the current membership
  (the key endpoint reports `needs_rotation`), and the server refuses any
  version whose envelopes don't cover the active membership exactly. This is
  what makes membership boundaries cryptographic: a newly added member holds
  envelopes only for post-join versions (and the server additionally never
  serves them pre-join messages at all), while a removed member is excluded
  from every later version, so post-removal messages are unreadable to them.
  Removed members keep their old envelopes — their own history stays readable,
  the same recoverability trade as everywhere else in this design.

## Wire formats

All blobs are standard base64 (with padding). The server validates shape and
length only. See `test_e2ee_interop.py` for executable PyNaCl equivalents.

| Field | Contents |
| --- | --- |
| `MessagingKeyBundle.public_key` | X25519 public key (32 bytes) |
| `MessagingKeyBundle.password_wrapped_secret` | `nonce(24) ‖ secretbox(privkey)` under `wrapKey` |
| `MessagingKeyBundle.password_wrap_salt` | Argon2id salt for `wrapKey` (16 bytes) |
| `MessagingKeyBundle.recovery_wrapped_secret` | `nonce(24) ‖ secretbox(privkey)` under the recovery key |
| `AccountKdf.auth_salt` | Argon2id salt for `authKey` (16 bytes) |
| `ConversationKey.wrapped_for_{low,high}` | `crypto_box_seal(conversation_key)` to each party's public key |
| `GroupKeyEnvelope.wrapped_key` | `crypto_box_seal(group_key)` to one member's public key |
| `DirectMessage.ciphertext` | `secretbox(body)` under the conversation key |
| `DirectMessage.nonce` | 24-byte nonce for `ciphertext` |
| `DirectMessage.key_version` | which `ConversationKey.version` encrypted it (0 = plaintext) |
| `GroupMessage.ciphertext` / `nonce` / `key_version` | same contract, under the `GroupKey` version instead |

Argon2id parameters are pinned per bundle (`kdf_opslimit=2`,
`kdf_memlimit=64 MiB`, libsodium "interactive") so a future parameter change
never orphans existing blobs.

## Enrollment and login

Enrollment is transparent — there is no separate "encryption passphrase" and no
setup screen.

- **New signup (password):** the signup form derives `authKey` in the browser
  and submits it plus `auth_salt` in place of the raw password, so the server
  never sees the real password even at account creation. The keypair itself is
  generated at first login after email verification (signup doesn't log the
  user in).
- **Login (password):** the login form fetches `/e2ee/login-params/` for the
  typed identifier, derives `authKey`+`wrapKey`, logs in via `fetch` with
  `authKey`, then unwraps and caches the private key. First login of a
  pre-E2EE ("legacy") account logs in with the raw password once, then upgrades
  the account to derived auth and shows the recovery key.
- **OAuth (Google/Discord):** no password exists at signup, so there is
  initially no password-wrapped copy. The keypair is generated in the
  background on the first authenticated page load and the recovery-wrapped
  copy is uploaded. After any login, an account with no usable password is
  prompted (skippable per session) to **set a password** — the set-password
  flow derives `authKey`/`wrapKey` in the browser exactly like a password
  signup, and when the device holds the decrypted private key it uploads a
  password-wrapped copy too. From then on a new device can unlock with the
  password instead of the recovery key. Until a password is set, a **new
  device requires the recovery key** — the irreducible floor without
  server-side escrow.

`/e2ee/login-params/` returns a deterministic decoy salt for unknown
identifiers so enrolled and non-existent accounts are indistinguishable.

## Recovery flows

- **Multi-device (password):** each device derives `wrapKey` from the password
  at login and unwraps independently. No key transfer needed.
- **Multi-device (OAuth):** the locked-device banner/dialog accepts the account
  password (once one has been set and a password-wrapped copy exists) or the
  recovery key, once per new device.
- **Password change (Settings → Security):** requires typing the current
  password (verified as whatever credential the server stores — raw for
  legacy, derived for enrolled accounts) and always rotates to derived auth
  with a fresh salt. A device holding the decrypted key re-wraps it under the
  new password in the same request; otherwise the old wrapped copy is flagged
  stale, exactly like an email-link reset. A "Forgot your password?" link
  points at the email reset flow.
- **Password reset (email link):** the reset never sees the old password, so
  the password-wrapped key copy becomes undecryptable. The reset-confirm page
  derives a new `authKey` (fresh salt) and the server flags the wrapped copy
  stale. On the next login from a device that still holds the cached private
  key, the client silently re-wraps it under the new password. A cold device
  falls back to the recovery key.
- **Lost everything (password + recovery key):** history is unrecoverable —
  this is inherent to real E2EE. The **Reset keys** action (Settings → Direct
  Messages) generates a fresh keypair; the user's old encrypted messages become
  unreadable to them, but each conversation partner keeps their own copy
  (old `ConversationKey` versions are retained), and new messages encrypt under
  a fresh conversation-key version.

## Opportunistic encryption

A message is encrypted **iff both participants have published a key bundle**.
Since enrollment happens automatically at login, conversations upgrade to
encrypted on their own within a login or two. Until the partner has ever logged
in post-deploy, messages fall back to plaintext (no padlock). Pre-E2EE messages
stay plaintext; only encrypted messages render a padlock.

## Operational caveats

- **JavaScript is required** for password login once an account is upgraded to
  derived auth (the app is already JS-dependent everywhere).
- **The Django admin login form breaks for upgraded staff** — it submits the
  raw password, but the stored hash is of `authKey`. Staff should log in via the
  normal site login (the session carries into `/admin/`). `manage.py
  changepassword` has the same caveat: it sets a raw password Django will hash,
  which won't match the derived-auth login flow — use it only to *downgrade* an
  account back to legacy auth (also delete the `AccountKdf` row).
- **Exports** include encrypted messages as ciphertext plus a note; a readable
  copy is available via the in-browser "Download transcript" action on the
  messages page (the only place plaintext is ever assembled).
